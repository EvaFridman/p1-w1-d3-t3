# p1-w1-d3-t3 · all, allSettled, race, any

> Тип: крепкая (статические методы Promise) · Репозиторий: один отдельный

> Подключить `.js` к HTML через `<script>`, отсмотреть в браузере. Сегодня только промисы — `async/await` будет завтра, его не используем.

## Задача

Когда запросов много и они не зависят друг от друга, ждать их по очереди — расточительство. Класс Promise даёт четыре статических метода, чтобы работать с пачкой промисов: дождаться всех, дождаться всех несмотря на сбои, взять первого пришедшего или первого успешного. Разбираешься, чем они отличаются, на живых запросах.

```
all / allSettled / race / any — много промисов
┌──────────────────────────────────────────────────────┐
│ all — ждёт всех, один упал → упало всё               │
│ allSettled — ждёт всех, отдаёт статус каждого        │
│ race — первый пришедший (успех ИЛИ ошибка)           │
│ any — первый УСПЕШНЫЙ                                │
└──────────────────────────────────────────────────────┘
параллельная загрузка · таймаут гонкой · устойчивость
```

## API

`https://jsonplaceholder.typicode.com` — ресурсы `/users`, `/posts`, `/comments`, `/todos`, `/albums`. Для заведомой ошибки: любой id вне набора (например `/posts/99999`) даёт 404, а несуществующий домен — сетевой сбой.

## Дано

```html
<div id="app">
  <button data-run="all">all</button>
  <button data-run="allSettled">allSettled</button>
  <button data-run="race">race</button>
  <button data-run="any">any</button>
  <pre id="out"></pre>
</div>
```

## Что нужно сделать (по релизам)

**Release 1 — all и параллельность.** Кнопкой `all` запросить параллельно три разных ресурса (например пользователей, посты и todo) и, когда все придут, вывести сводку: сколько записей в каждом. Замерить время выполнения. Затем сравнить: запросить те же три ресурса последовательной цепочкой и замерить снова — показать в `#out`, что параллельно заметно быстрее. Потом подмешать в пачку заведомо провальный запрос и убедиться, что весь `all` отклоняется, а успешные результаты остальных теряются.

**Release 2 — allSettled как страховка.** Та же пачка с одним заведомо провальным запросом, но через `allSettled`. Показать, что результат приходит по каждому запросу со своим статусом: успешные — с данными, провальный — с причиной. Вывести в `#out` таблицу «запрос → успех/ошибка». Сформулировать правило: когда нужен `all`, а когда `allSettled`.

**Release 3 — race и any.** Кнопкой `race` устроить гонку запроса с таймаутом: собственная функция-таймаут (промис, отклоняющийся через N мс — возьми из прошлой задачи идею задержки) против реального запроса. Показать оба исхода: при большом таймауте побеждает запрос, при крошечном (например 1 мс) — таймаут, и это уходит в обработчик ошибки. Кнопкой `any` смоделировать «зеркала»: пачка из заведомо провального запроса и рабочего — показать, что `any` игнорирует сбой и отдаёт первый успешный. В `NOTES.md` — таблица из четырёх методов: что ждёт, когда падает, для какой задачи.

## Ожидаемый результат

- Параллельная загрузка через `all` заметно быстрее последовательной цепочки тех же запросов (видно по замеру); один сбойный запрос обрушивает весь `all`, и данные успешных недоступны.
- `allSettled` в той же ситуации доводит дело до конца и отдаёт статус по каждому запросу — видно, кто успешен, кто нет и почему.
- В гонке с большим таймаутом выигрывает запрос, с крошечным — таймаут (и это трактуется как ошибка); `any` возвращает первый успешный результат, игнорируя сбойный.
- В `NOTES.md` — сравнительная таблица четырёх методов с областью применения.

## Критерии приёмки

* Показана и замерена разница параллельной загрузки (`all`) и последовательной цепочки.
* Продемонстрировано поведение `all` при сбое одного промиса и как `allSettled` это решает (статус по каждому).
* `race` использован для таймаута (оба исхода показаны), `any` — для «первого успешного» среди сбойных.
* В `NOTES.md` — таблица различий четырёх методов и когда какой брать.
